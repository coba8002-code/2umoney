# 기능적 고도화 기획서 (Functional Enhancement Roadmap)

> 대상: 베리어프리 접근성 검증·자동보정 플랫폼
> 기준: 현재 구현된 Phase 1(코어 룰셋·색보정·Figma 플러그인·웹 플레이그라운드) 위에서의 **기능 고도화**.
> 목적: "검사 정확도 ↑ · 자동 보정 범위 ↑ · 적용 워크플로 ↑" 세 축으로 제품 가치를 끌어올린다.

---

## 0. 현재 역량과 한계 (정직한 출발점)

**할 수 있는 것**
- 8개 자동 룰: 텍스트/비텍스트 대비, 터치 타깃, 글자 크기, alt, 레이블, 포커스 표시
- 스타일 유지 색 보정(`nearestPassingColor`): 색상·채도 고정, 명도만 조정 + 자기검증
- Figma 노드 → `A11yNode` 정규화(유효 배경색 상속), 보정 적용(색/크기), Variables 토큰 바인딩
- 출처 구분 리포트(auto/ai-assisted/manual), JSON/텍스트 출력

**현재 한계 (고도화 대상)**
| # | 한계 | 영향 |
|---|---|---|
| L1 | 단색(SolidPaint)만 대비 계산 — **그라데이션·이미지 위 텍스트·반투명** 미지원 | 실제 디자인의 상당수 케이스 누락 |
| L2 | 유효 배경색이 `opacity≥0.5` 단순 규칙 | 겹친 반투명 레이어에서 부정확 |
| L3 | Figma엔 alt/label/포커스 **의미 정보가 없어 휴리스틱**(레이어명 추론) | 오탐/미탐 가능 |
| L4 | `heading.structure` 등 구조 룰은 HTML 전용이라 **미구현** | 문서 구조 검증 공백 |
| L5 | AI 계층(`LlmProvider`)은 **인터페이스만**, 실제 alt 생성·문맥 평가 없음 | 의미 항목은 수동에 의존 |
| L6 | 보정이 **노드 단위** — 팔레트/토큰 전체 일관성 미고려 | 보정 후 색 충돌·중복 |
| L7 | 키보드 순서·읽기 순서·모션·확대 등 **다수 KWCAG 항목 미커버** | "일부만 검사" 한계 |

## 1. 고도화 트랙 개요

```
Track A. 검사 엔진 정확도·범위   (코어 신뢰도)      ← 가장 먼저
Track B. 자동 보정 지능화        (차별화 핵심)
Track C. AI 계층 실구현          (의미 항목 자동화)
Track D. HTML/URL 파이프라인     (Phase 2, 시장 확대)
Track E. 대시보드·협업·CI        (지속 사용·B2B)
Track F. 키오스크 빌더           (Phase 3, 공공)
```

---

## 2. Track A — 검사 엔진 정확도·범위 ⭐ 최우선

### A1. 그라데이션·이미지·반투명 대비 (해결 L1, L2)
- **무엇**: 배경이 그라데이션/이미지/반투명일 때 **최악 지점(worst-case) 대비**를 계산.
- **구현**: `packages/core`에 `resolveEffectiveBackground()` 고도화. 그라데이션은 stop 색들 중 텍스트와 가장 대비가 낮은 색으로 평가, 반투명은 알파 블렌딩(`over` 합성)으로 실효색 산출. Figma 어댑터의 `resolveEffectiveBg`를 이 코어 함수로 이관.
- **난이도**: 중 · **가치**: 높음 (실사용 정확도 직결)
- **수용 기준**: 그라데이션 위 텍스트가 최저 대비 지점 기준으로 fail 판정되고, 알파 0.5 오버레이가 블렌딩 결과로 평가됨.

### A2. 자동 룰 확장 (해결 L4, L7)
신규 룰을 `rules-data` + `core/rules`에 추가(각각 `auto`/`manual` 분류 명확화):
- `text.lineHeight` (행간), `text.letterSpacing` (자간) — 가독성
- `link.identifiable` (색만으로 구분 금지 — 밑줄/아이콘 동반)
- `input.labelAssociation` (입력-레이블 명시 연결)
- `motion.autoplay` (자동재생/모션 경고 → manual 가이드)
- `heading.structure` (HTML 도입 시 자동, Figma는 레이어 계층 휴리스틱)
- **난이도**: 룰당 소~중 · **가치**: 커버리지 ↑ → "일부만 검사" 한계 완화
- **수용 기준**: KWCAG 2.2 33개 항목 중 기계판정 가능 항목 커버율을 리포트에 명시.

### A3. 판정 신뢰도(confidence) 도입 (해결 L3)
- **무엇**: 각 `Finding`에 `confidence: 'high'|'medium'|'low'`와 근거를 부여. 휴리스틱 기반(Figma alt/label 등)은 자동으로 낮은 신뢰도 + "수동 확인 권장" 플래그.
- **구현**: `Finding`에 필드 추가, 엔진에서 룰별 신뢰도 부여, UI에서 배지 표시.
- **난이도**: 소 · **가치**: 신뢰·법적 안전(과탐 방지).

## 3. Track B — 자동 보정 지능화 ⭐ 차별화 핵심

### B1. 팔레트·토큰 인지 보정 (해결 L6)
- **무엇**: 색을 개별로 고치지 않고, **디자인 시스템 팔레트/Variables 안에서 통과하는 가장 가까운 색**을 우선 선택. 팔레트에 없으면 신규 색을 제안하되 토큰화 권고.
- **구현**: `nearestPassingColor`에 `candidatePalette?: string[]` 옵션 추가 → 후보 중 (통과 ∧ 최소 ΔE) 선택, 없을 때만 연속 탐색. Figma 어댑터가 로컬 Variables를 후보로 주입.
- **난이도**: 중 · **가치**: 매우 높음 (보정이 디자인 시스템과 충돌하지 않음).

### B2. 다목표 보정 옵션 (AA/AAA, 대비 vs 색차 최소화)
- **무엇**: 사용자가 목표를 선택: ① 최소 변경(현행) ② AAA(7:1) ③ 색맹 안전(色 구분 강화).
- **구현**: `FixSuggestion`에 보정 변형(variants) 배열. UI에서 미리보기 후 택1.
- **난이도**: 중 · **가치**: 높음.

### B3. 색각 이상 시뮬레이션 + 검증
- **무엇**: 적/녹/청색약 시뮬레이션을 거쳐 "색만으로 정보 전달" 위반 탐지 및 보정 검증.
- **구현**: `core/color`에 색각 변환 행렬 추가, `link.identifiable`/대비 룰과 연계.
- **난이도**: 중 · **가치**: 차별화 + 실효성.

### B4. 일괄 보정 트랜잭션 + 되돌리기
- **무엇**: "전체 자동 보정"을 하나의 트랜잭션으로 적용/롤백. 적용 전 **diff 미리보기 일괄 화면**.
- **구현**: Figma `code.ts`에 변경 스냅샷 저장, UI에 일괄 diff 패널.
- **난이도**: 중 · **가치**: 신뢰(가드레일 "사람이 수락 후 적용"과 부합).

---

## 4. Track C — AI 계층 실구현 (의미 항목 자동화)

### C1. 대체텍스트 생성 (`suggestAltText` 실구현)
- **무엇**: 이미지+주변 문맥을 멀티모달 LLM에 전달해 alt 초안 + confidence 생성. **장식/정보 구분**도 제안.
- **구현**: 스펙의 `LlmProvider` 인터페이스 구현체(Claude 등) + 프로바이더 추상화. Figma는 노드 래스터라이즈(`exportAsync`) 후 전송(네트워크 권한은 Phase 2에서 `allowedDomains` 확장).
- **가드레일**: 결과는 항상 **사람이 수락**, "AI 보조" 출처 표기 유지.
- **난이도**: 중 · **가치**: 높음 (critical 항목 자동화).

### C2. 문맥 판단형 항목 1차 평가 (`assessContextual`)
- **무엇**: 링크 텍스트 적절성, 레이블-기능 일치 등 맥락 항목을 AI가 1차 평가 → `manual` 가이드를 구체화.
- **난이도**: 중 · **가치**: 수동 항목의 검토 시간 단축.

### C3. 보정 경계 강제(코드 룰 우선)
- **원칙 유지**: 색·레이아웃은 **결정론적 알고리즘**, AI는 텍스트/판단 보조에만. (환각으로 인한 디자인 훼손 방지 — 스펙 7.3.)

## 5. Track D — HTML/URL 검사 파이프라인 (Phase 2)

### D1. 헤드리스 수집 + axe-core 결합
- **무엇**: URL/HTML 업로드 → Playwright로 DOM·계산 스타일 수집 → 자체 룰셋 + `axe-core` → `ScanResult`.
- **구현**: `packages/api`(Fastify) + `html-adapter`(DOM→`A11yNode`). 코어 룰은 그대로 재사용(룰셋 공유 원칙).
- **난이도**: 중상 · **가치**: 시장 확대(웹 전반).

### D2. 코드 자동 보정 + 검증 루프
- **무엇**: 색은 `nearestPassingColor`로 CSS 수정, 구조는 ARIA/label/alt/focus 보강(비가시 우선) → **재스캔 통과 확인 후 diff 반환**.
- **난이도**: 중상 · **가치**: 높음.

### D3. REST API (스펙 7.4 구현)
`POST /v1/scan`, `POST /v1/fix`, `GET /v1/report/:id`, `POST /v1/figma/scan` — 플러그인↔백엔드 연동.

---

## 6. Track E — 대시보드·협업·CI (지속 사용 / B2B)

- **E1. 리포트 히스토리·추세**: 스캔 결과 저장(PostgreSQL), 통과율 추이·회귀 감지.
- **E2. PDF/공유 리포트**: `reportToText`/`buildReport` 기반 PDF·공유 카드(PNG). 감리·보고용.
- **E3. CI 게이트(GitHub Action)**: PR 디자인/코드에 접근성 린트 → 임계치 미달 시 코멘트. (본 저장소의 Pages 워크플로 패턴 재사용.)
- **E4. 디자인↔코드 동기화**: Figma Variables ↔ CSS 토큰 매핑으로 보정 결과를 코드까지 반영.

## 7. Track F — 키오스크 빌더 (Phase 3)

- NIA 배리어프리 키오스크 디자인 시스템(공공누리 1유형, 출처표시)을 토큰/컴포넌트 라이브러리로 구조화.
- 입력(업종·브랜드 색·로고) → 접근성 검증된 키오스크 화면/HTML 생성 → Phase 1/2 엔진으로 자동 검증.
- 키오스크 특화 강화: 큰 터치 타깃(≥44, 권장 ≥48), 고대비 모드, 음성/저시력 모드 프리뷰.

## 9. 추천 — 다음 스프린트 (2주 가정)

**"코어 신뢰도 + 차별화 보정" 묶음** (Track A·B 우선)
1. **A1** 그라데이션/반투명 유효 배경색 + 최악지점 대비
2. **A3** `Finding.confidence` 도입 + UI 배지 (휴리스틱 항목 저신뢰 표시)
3. **B1** `nearestPassingColor` 팔레트/토큰 후보 지원 + Figma Variables 연동
4. **B4** 일괄 보정 diff 미리보기 + 롤백

→ 모두 **테스트 우선**(코어 케이스·경계값), 기존 가드레일(자동/AI/수동 구분, 사람 수락) 유지.

> 이 묶음은 새 인프라(백엔드·AI 네트워크) 없이 `packages/core`와 Figma 플러그인 안에서 완결되어, 리스크가 낮고 사용자 체감이 큽니다. AI(C)·HTML(D)·대시보드(E)는 그 다음 단계로 확장합니다.
